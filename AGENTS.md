## Architectural Patterns

- **Frontend routing: shared app-shell layout route with a shared chrome.** Every page is a React Router route nested under one pathless layout route rendered by `AppShell` (`frontend/src/AppShell.jsx`, wired in `frontend/src/main.jsx`). The shell owns the persistent left sidebar (the `Primary` nav — Dashboard, Trade market and **Player stats** on top, then the roster links grouped under collapsible **Active market** and **Inactive market** section headers (`SideGroup`, in-memory open state, links hidden via `.side-group.is-collapsed`)), a shared top navbar and a shared footer (both in `frontend/src/Chrome.jsx`), and renders the active page through an `<Outlet />` between them, so the chrome stays mounted while only the content area swaps. The shell owns the market poll and the Step/Start/Pause/Reset actions and exposes `{ market, connected, ready, pending, actionError, runAction }` via the outlet context (`useOutletContext`); the dashboard and trade-market pages read market state from there rather than polling `/market` themselves. Each page still owns its own data polling loop — a `setTimeout(0)` + `setInterval` pair cleaned up on unmount, StrictMode-safe — and renders only its `<main className="main">` (no per-page topbar/footer); `TradersPage` and `DepartedTradersPage` are the reference shape. Active nav uses `NavLink`'s `is-active` modifier, styled by shape and weight rather than color alone. There is no Player modal — the player surface lives on the dashboard (`PlayerPanel`), the persistent worth block in the sidebar, and a standalone **Player stats** page (`/player-stats`, `PlayerStatsPage`) that resolves the player id and reuses `ParticipantDetail`.
- **Entity roster pages navigate to standalone detail pages.** Traders and Companies each have a roster page (`TradersPage`, `CompaniesPage`) that shows one server-paged table — name search, sortable numeric columns, and a type (traders) / industry (companies) filter, all sent to `GET /participants/paged` / `GET /companies/paged` (`page`/`pageSize`/`search`/`sort`/`sortDir` plus `type`/`industryId`; the plain array `GET /participants` and `GET /companies` endpoints stay for the dashboard map and its aggregates, and `CompanyResponse` now also carries a `CurrentRating` from a latest-rating-per-company lookup). The tables (`TradersTable` / `CompaniesTable`) are presentational — sort is controlled by the page via `onToggleSort`, and they render without the `.tbl-scroll` inner scrollbar so the page scrolls instead. Clicking a row navigates to a standalone detail route — `/traders/:id` (`TraderDetailPage`) or `/companies/:id` (`CompanyDetailPage`) — each wrapping `ParticipantDetail` / `CompanyDetail`, the single source of that entity's full detail (identity, its own chart, related news, ratings; owns polling keyed on the route id; `.detail-stack` layout). Every deep-link points at these routes (`CompanyModal`, `CompanyDetail`'s shareholders list, and `ParticipantDetail`'s fund/member links). The dashboard is Market map + player block + order book only: the player block (`PlayerPanel`) shows balances and performance, then presents its detail views as a **tab strip** after the performance block (reusing the `.tabs`/`.tab`/`.tabpanel` chrome, with a roving-tabindex tablist like the order book) — a client-paged/sortable **active assets** table, a **by-industry** portfolio breakdown, a **companies-needing-attention** table (`GET /participants/{id}/companies-attention`, scoped to the participant's held companies, flagging a recent price slide, a bad news/crisis hit, a standing High/Extra rating, or a recent reverse merge), **open orders**, a **total-worth chart** (`GET /participants/{id}/worth-history`, via `LineChart`), **cash movements** (`GET /participants/{id}/money-transactions`, tone/label from the shared `cashMovements.js` also used by `ParticipantDetail`), and **loans** — the in-panel Place Order form is gone (the player trades from the company view). The by-industry breakdown has no endpoint: it is a pure client-side join of the already-loaded holdings and companies (which carry the industry), grouped per industry (companies held, shares, value, % of portfolio, P/L) by the shared `industryHoldings.js` helper and rendered by the shared `IndustryHoldingsTable`, whose industry cell links to `/industries/:id` (a held company missing from the live list buckets as a non-linked "Unknown industry"). The same pair is a **Portfolio by industry** panel in `ParticipantDetail`'s `.detail-stack`, so it shows on every trader detail page too. Client-side paging+sort for these small per-player tables is `useClientTable` with the shared `SortHeader`/`Pager` (`TableControls.jsx`). The order book's Buy tab carries an All / **Player owns** filter (default Player owns → buy orders for companies the player holds).
- **The market map and industries map share one `Treemap`.** `Treemap` (`frontend/src/Treemap.jsx`) owns the squarified layout (`squarify`) and tile rendering — area by `value`, colour by `changePct` with a glyph and signed percent so it is never colour-only — and takes items that each carry their own `title`/`ariaLabel`. `MarketMapPanel` (`frontend/src/MarketMapPanel.jsx`) feeds it companies behind a filter bar (industry multiselect, capitalisation quartile band, player-shares, auditor risk — the risk band reads `CompanyResponse.CurrentRating`) and renders the two latest posts (`LatestNews.jsx`, company-linked via `onSelectCompany`) below the treemap. It is reused on both the dashboard (tiles open a `CompanyModal`) and the **Trade market** page (`/trade-market`, `TradeMarketPage`, in the sidebar right after Dashboard), which pairs it in a tab with the orders-per-cycle chart (`OrdersActivity.jsx`) and shows the two latest posts; there tiles navigate to `/companies/:id`. The manual news composer (`AddNewsModal.jsx`) lives on the News page (`/news`, `NewsPage`), a server-paged (`GET /news/paged`) feed of full newswire cards (`.news-feed`) that show each post's body inline and open `NewsModal` from the headline for scope/industry detail; the shared `NewsImpact` renders a post's market effect and, when given an `onSelectCompany`, a clickable target-company name. `NewsPost.Category` (`NewsCategory`: `General`/`CompanyClosed`/`StockSplit`/`StockMerge`, surfaced on `NewsPostResponse.Category`) is the typed way to flag a corporate-action post rather than string-matching its `Scope = None` title — `newsCategory.js` maps it to a card treatment reused by every newswire card (`LatestNews.jsx` and the News feed): a delisting gets the crisis red frame (`--down`/`--down-soft`), a split/merge the cyan info frame (`--info`), each with a text tag so the flag is never colour-only. `IndustriesPage` (`/industries`, in the sidebar) feeds it industries aggregated in the browser from `GET /companies` (total worth summed per industry), with a non-paginated industries table below it and a per-industry company modal (`IndustryModal`, companies linking to `/companies/:id`); its last-cycle change is the same forward-only capitalisation diff the market map uses (0 until a cycle advances after load), so no backend endpoint backs it. The two order forms (`OrderForm` in `CompanyModal`, `TradeOrderModal`) share `PercentButtons` for quantity presets (% of owned shares on a sell, of cash-affordable quantity on a buy) and price presets (± off the current price).
- **Sector sentiment has two separate market channels.** Each sector takes a slow, bounded mood walk with mean reversion, reacting to crises, science investigations, and news. Its fixed `SectorBeta` and clamped sentiment jointly scale only crisis, science, and scoped-news shocks: neutral sentiment leaves beta active, while the mood adjustment uses beta × (1 ± clamped mood) to cushion or amplify the move. Ordinary matching never uses this shock adjustment; a separate demand channel nudges AI buy or sell demand toward favoured or shunned sectors. Scoped news impacts land after matching at the advance boundary and are marked with `ImpactAppliedInCycleId`, so they inform the next decision cycle without trading against the settled book. `SectorSentimentSnapshot` follows the shared 500-cycle retention policy into its archive twin.
- **Worth snapshots cover every trader.** `MarketService.WriteWorthSnapshotsAsync` records one `ParticipantWorthSnapshot` (cash + holdings value) per participant per completed cycle, read back chronologically by `GET /participants/{id}/worth-history` to chart total worth over time. Snapshots are captured going forward only — traders that predate the first write start with an empty chart. `PriceSnapshot.Capitalization` follows the same forward-only rule: every snapshot-creation site (matching, `MarketImpactService`, `StockSplitService`, seed) stamps `price × IssuedSharesCount` so the company capitalisation chart stays continuous across a split (where price falls but the share count rises); snapshots predating the column are null and the chart skips them.
- **Latest price is a max-Id lookup, and the decision pass batches its writes.** Every per-cycle price read goes through the shared `PriceSnapshotQueries.LatestPriceByCompanyAsync` (in `Services`), which finds each company's latest price by its max snapshot `Id` and reads `AsNoTracking` — it replaced the identical `PriceSnapshots.ToListAsync()` + in-memory group that had been copied into `MarketService` and every per-cycle service, each of which re-materialised the whole snapshot history (and, being tracked, bloated the change tracker) on every call. `MarketService.GenerateDecisionsCoreAsync` builds the price map once and threads it into `PlaceOrderCoreAsync`/`DebtAllowanceAsync` (optional `priceByCompany`) so a buy never re-reads prices, and stages every order behind one `SaveChangesAsync` (`deferSave: true`) instead of saving per order — safe only because `RuleBasedDecisionEngine` emits at most one order per trader and skips companies the trader already has an open order in, so no staged order races the per-order validation. Order-book reads lean on the `Orders(Status)` and `Orders(ParticipantId, CompanyId, Type, Status)` indexes; price reads on `PriceSnapshots(CompanyId, Id)`.
- **Aged rows are archived to cold-storage tables on a retention window.** `MarketService.ArchiveAgedRowsAsync`, run at the end of `AdvanceCycleCoreAsync` inside the advance transaction, bulk-moves `PriceSnapshot`/`MoneyTransaction`/`ParticipantWorthSnapshot`/`SectorSentimentSnapshot` rows whose cycle is older than `Archive:RetentionCycles` (default 500, `ArchiveOptions`, opt-out `Enabled`) into the matching `PriceSnapshotArchive`/`MoneyTransactionArchive`/`ParticipantWorthSnapshotArchive`/`SectorSentimentSnapshotArchive` tables — plain column copies with no navigations that preserve the original `Id`, moved by an `INSERT ... SELECT` plus `ExecuteDeleteAsync` keyed on `CreatedInCycleId` (indexed on all four source tables). The running simulation and the API read only the live tables, so history charts/ledgers show just the retained window and the retention must stay above every live look-back (the long-range price window, the auditor history window, the recent-dividend window — all far under 500). The `AddArchiveTablesAndIndexes` migration backfills the pre-existing backlog once so the first post-deploy cycle is not slow; `ResetDemoMarketCoreAsync` clears the archive tables and their sequences alongside the live ones. Because rows move within the same database file, the file does not shrink until a manual `VACUUM` — the win is a small working set, not a smaller file.
- **Holdings are quantities, not per-share rows.** A participant's position in a company is a single `Holding` row (`ParticipantId`, `CompanyId`, `Quantity`, `AverageCost` = weighted-average price paid); there is no per-share entity and no `OrderShare` join. The issuer's unsold float is implicit — `IssuedSharesCount − Σ Holdings.Quantity` for the company — and is listed on the null-participant sell order. `MatchingEngine.ExecuteFill` decrements the seller's holding and blends the fill price into the buyer's `AverageCost` (a sold-out holding keeps a zero-quantity row rather than being deleted — see `ReduceHolding` — and every holdings read filters on `Quantity > 0`, so any code inserting a `Holding` must exclude the participant by row existence, not by `Quantity > 0`, to avoid the unique-key clash); a company-float sale (`seller == null`) creates/adds to the buyer's holding with no seller decrement. A sell is validated and sized against `Holding.Quantity` minus the seller's already-listed open-sell quantity — the per-cycle services (`BankruptcyService`, `CollectiveFundService`) carry this as an in-memory `available`/listed tally keyed on `(participantId, companyId)` that replaced the old committed-`ShareId` set, decrementing it as they list forced/idle sells. Holdings value everywhere is `Σ Quantity × latest price`; cost basis (`HoldingResponse`/`ShareholderResponse`) is `Quantity × AverageCost`. This keeps stock splits/merges pure arithmetic on holdings and orders rather than multiplying millions of rows.
- **Price stability: stock splits, reverse merges + a dividend ceiling.** Two mechanisms keep the closed economy from inflating without bound. `StockSplitService` is the deterministic member of the per-cycle service family — it takes no `Random` and draws nothing — and splits a company `SplitRatio`-for-1 once its latest price crosses `SplitPriceThreshold`: `IssuedSharesCount` and every `Holding.Quantity` scale up while the price and `AverageCost` scale down, so market cap and each holder's worth are unchanged and only the denomination shrinks. The issuer float is re-denominated in place (quantity up, limit down) because cancelling it would strand the unsold supply; every participant order for the company — the human player's included — is cancelled instead (buys release their reserved cash) so the book re-forms at the new price. Cancelling the player's orders is a deliberate exception, scoped to these denomination changes, to the otherwise-universal rule that the market never auto-cancels the player. Each split also stages an impact-free `NewsPost` (`Scope = None`, no industries or target company) announcing it. The service runs first in the pre-match window so bankruptcy/funds/exits read the post-split state, and a `MaxIssuedShares` guard stops splitting before the 32-bit quantity fields could overflow. The same service is symmetric at the low end: a price below `MergePriceThreshold` merges the stock `SplitRatio`-to-1 — `IssuedSharesCount` and every `Holding.Quantity` scale down while price and `AverageCost` scale up — with the same float re-denomination, participant/player order cancellation, cap-preserving snapshot, and `Scope = None` news. A merge also stamps `Company.LastMergedInCycleId`, so recency (e.g. the companies-needing-attention query) can be tested without string-matching the `Scope = None` news post. Unlike a split, a merge divides integer share counts, so it floors each holding to a whole share and drops the sub-share remainder (a holder of fewer than `SplitRatio` shares is zeroed) rather than paying cash-in-lieu; `IssuedSharesCount` is re-derived as the floored holdings plus the floored implicit float so the `IssuedSharesCount − Σ Holdings` identity stays exact, and a holder's worth can tick down by under one merged share. A `MinIssuedShares` floor mirrors the overflow guard, and a still-cheap penny stock simply merges again next cycle until it clears the threshold. Separately, `MarketService.PayDividendsAsync` sizes each paying company's payout pool as a random rate (`MinDividendRate`–`MaxDividendRate`) of its capitalisation, shared evenly across `IssuedSharesCount` — so per share it is `rate × price` and a split cuts the per-share dividend proportionally on its own, with no dividend code in `StockSplitService`. A per-company cash ceiling (`MaxDividendCashPerCompanyPerPayout`) still compresses the rate so new money — dividends have no counterparty — cannot track an ever-rising market cap; the ceiling is keyed on total owned value (split-invariant), so it composes with splits. Payment is also gated per company per window: on each due window (the market-wide `NextDividendCycleNumber` schedule still decides *when* a window opens) a company pays with `StableCapitalizationChance` while its capitalisation has held within `CapitalizationStabilityThreshold` of its last-window baseline, and only `VolatileCapitalizationChance` once it has moved past it — thinning payouts during turbulent stretches. The baseline is `Company.LastDividendCapitalization`, refreshed for every priced company each window (a null baseline, or the first window, counts as stable); because capitalisation is split-invariant, a split never spuriously flips a company into the volatile branch. Draw discipline for scripted-`Random` tests: companies are walked in ascending `Id` order, each priced company drawing one `NextDouble` for its pay roll and — only if it pays — one more for its rate.
- **Per-cycle simulation services.** Rules that mutate the market each tick (industry sentiment, bankruptcies, collective funds, market exits, stock splits, share emissions, auditor ratings, company lifecycle) are sealed services taking `(AppDbContext, IOptions<TOptions>, IOptions<RandomChanceRatesOptions>, Random)` — `StockSplitService` omits both the `Random` (it is fully deterministic) and the chance-rates options (it draws nothing), and `AuditorService` and `CompanyLifecycleService` also take `MarketImpactService` (the latter to apply the large-cap price cut in place of a closure) — gated by an opt-in `Enabled` option, and invoked from `MarketService.MaintainOrdersCoreAsync` before the decision pass. They stage changes on the shared context and let the caller save; each documents its random-draw discipline in a top comment so a scripted `Random` stays reproducible in tests. `BankruptcyService` is the canonical template. The sentiment walk runs before the remaining pre-match sequence so decisions read a settled sector mood. That order then matters: `VolatilityHaltService` runs first (reading only prior-cycle closes, before any this-cycle price move, so a same-cycle deliberate cut cannot self-trip its down-halt), then `StockSplitService`, then `ShareEmissionService`, then `CompanyLifecycleService` (the three supply-side corporate actions, each with its own save), then liquidation/bankruptcy/funds, then `MarketExitService`, then `ConcentrationCapService` (a relative-weight price cut, so bankruptcy/funds/exits read pre-cut prices), and finally `AuditorService` last of all (its own save) so bankruptcy, funds, and exits read pre-audit prices while this cycle's decisions and matching react to the fresh ratings and cuts. `MarketExitService` is the only service that hard-deletes a `Participant` row; both its exit gates (starvation and fund-loss) exclude current fund members, so a live `CollectiveFundParticipant` is never orphaned, and `CollectiveFundService` still defensively drops any membership whose participant has gone missing rather than dereferencing it.
- **Every chance/probability and random-magnitude value lives in one config.** `RandomChanceRatesOptions` (bound to the `RandomChanceRates` config section, seeded in `appsettings.json`) is the single home for the market's tunable randomness, split into three nested groups: `EventTriggerChances` (probabilities that gate whether a random event fires — fund join/open/leave/switch, bankruptcy ramp cap and per-debt rate, company-appearance tiers, auditor discovery and buy-revision chances, exit rolls, the crisis/science trigger-chance step-per-cycle, share-emission chance, dividend pay chances, and the two automated-news probabilities), `ChanceModifiers` (values that scale or shift a base chance — the crisis bankruptcy/auditor/exit multipliers, the crisis science and news-suppression factors, the decision-engine crisis buy suppression, and the company-closure appearance boost), and `RandomMagnitudeBands` (the min/max bounds a draw picks its size from — dividend rate, emission rate, crisis and science industry-impact percents, global-crisis sector share). Each property's default equals the value it replaced, so injecting the options changes nothing until a config value is overridden. Every per-cycle service (and `RuleBasedDecisionEngine`, and `MarketService` for dividends) reads these through the injected `IOptions<RandomChanceRatesOptions>` rather than a private `const`; structural fractions and thresholds that are not chance/magnitude values (debt limit, sell-down fraction, protection-cap fraction, quiet-cycle windows, the engine's signal-pull weights) stay as service-local consts. When adding a new tunable chance or magnitude, add a commented property to the right group and bind it — do not reintroduce a hardcoded probability in a service.
- **The company population is a live roster that appears and delists.** `CompanyLifecycleService` runs each cycle right after share emission and does two independent things. Nothing closes during the market's first 100 cycles (a grace period so fresh listings can establish). Closure is deterministic (draws nothing): past that window a live company qualifies when its price fell in at least 16 of the last 20 recorded per-cycle closes, or its three most recent `CompanyRating` rows are all `High`/`Extra`; at most one company closes per cycle — the worst performer by recent price change when several qualify — and when the market is full at the `MaxCompanies` (300) cap with nothing failing on its own, the single worst *sub-threshold* performer is force-delisted to make room. A company whose total worth (latest price × `IssuedSharesCount`) is at least **0.5% of total market capitalisation** (the sum of every live company's worth) is never closed: a would-be-closed large-cap instead has its share price cut **60%** via `MarketImpactService.ApplyImpactAsync` and stays listed (this is not a closure — it frees no slot and does not feed the appearance boost). Because the threshold is relative, in a market of up to ~200 comparably-sized companies most are protected and only the smallest tail delists, and one 60% cut typically drops a near-threshold company below the line within a cycle or two so it can then close normally. The force-delist valve likewise only ever removes sub-threshold companies, so a large-cap is never delisted just to make room. Closing cancels every order for the company (buy reservations released as `Release`), zeroes every holding **with no payout** (holders recover nothing — a deliberate closed-economy choice, so no `MoneyTransaction` is written), stamps `Company.ClosedInCycleId`/`ClosedAt`, and posts `Scope = None` news. Appearance is the only random part: the base chance to mint a new company each cycle is a population tier keyed on the live-company count — 10% below 50 companies, 5% below 100, 1% at or above — blocked at the cap, and each delisting since the last listing adds a fixed +0.25 on top (`Market.CompanyClosuresSinceLastAppearance`, incremented per close and reset on each listing) so a shrinking population refills quickly and a delisting triggers a fast same-cycle replacement. Because the base chance is always positive, a roll is drawn every cycle the market is below the cap (so closure code that scripts a `Random` in tests must account for the extra roll); a listing draws a share count and price in the seed's bands (a random starting capitalisation is their product), a random industry and name (`DemoMarketNames.PickOneCompany`), records `Company.CreatedInCycleId`, lists the whole float as one company-originated `Open` sell, stamps an initial `PriceSnapshot`, and posts `Scope = None` news. A closed company is **soft-deleted** — the row survives so history, ratings, news `TargetCompanyId`, and the `/companies/:id` deep-link still resolve — and is excluded from the live market by an explicit `ClosedInCycleId == null` filter at each company-iteration site (the three supply-side services, dividend payout, the decision-pass `CompanyQuote` universe, and `BuildCompanyResponsesAsync`); no global EF query filter is used, so historical name lookups keep seeing closed companies. Closed companies surface on the Closed Companies page (`ClosedCompaniesPage`, `GET /companies/closed`) and carry an `IsClosed` flag on the detail response for a delisted banner.
- **Auditors are a standalone entity, not a `Participant`.** An `Auditor` never trades, holds, borrows, or carries a balance, so it is its own table rather than a `ParticipantType` — this keeps it out of every trading switch site (bankruptcy, exit, funds, decisions, ageing, liquidation) with no exclusion guards. `AuditorService` reviews one company per auditor per cycle: it seeds/backfills auditors at `ceil(companyCount × 0.05)` (drawing nothing), picks an eligible company by capitalisation-weighted random with a `1/eligibleCount` floor, and skips any company rated within the 15-cycle safe period. A company's per-cycle compound price move over the last 10 cycles classifies it stable (`Low`) or big-move (`High`); a discovery roll (2% stable, 10% big-move) escalates either to `Extra`, which drops the price 15–35% via `MarketImpactService.ApplyImpactAsync(..., cancelStaleOrders: false)` — `false` because the personality-weighted buy revision (base 50% High / 70% Extra, ± risk-profile and temperament deltas, Player and bankrupt owners skipped) replaces the blanket cancel so buys are never cancelled twice. `Low` posts no news; `High` posts `Scope = None`; `Extra` posts `Scope = Company` with the drop. Each `CompanyRating` row is one verdict; the newest inside the safe period is the current rating and the one before it gives the change shown on the company page.
- **A crisis is an active window, not a one-tick shock.** `CrisisService.MaybeTriggerForCycleAsync` still fires from `AdvanceCycleCoreAsync` and drops its hit sectors once, but each `Crisis` now stamps `TriggeredInCycleNumber` + `DurationCycles` (a per-crisis random draw taken right after the content draws — Local 5–15, Global 15–25) and stays *active* while the current cycle is in `(TriggeredInCycleNumber, +DurationCycles]`. `CrisisService.GetActiveCrisisAsync(cycleNumber)` returns the most-recent active crisis (or null), resolved once per phase in `MarketService` and threaded into the services that behave harsher during a crisis: `AuditorService` triples its issue-discovery chance (`CrisisIssueChanceMultiplier`, clamped to 1), `BankruptcyService` doubles every trader's computed probability (`CrisisBankruptcyMultiplier`, clamped to 1), `MarketExitService` scales every trader's quit chance — both the starvation and fund-loss gates — by `GlobalCrisisExitMultiplier` (5) in a global crisis and `LocalCrisisExitMultiplier` (2) in a local one (clamped to 1), `ScienceInvestigationService` halves its trigger chance and `NewsService` suppresses half of would-be price-*lifting* automated posts (both because a crisis should damp good news), and `RuleBasedDecisionEngine` reads a new `DecisionContext.CrisisActive` flag to drop a conservative *and/or* low-risk trader's buys by 15% per matching trait (stacking, ≈28% for both). All probability-scaling stays draw-discipline-safe (scale the threshold, keep the one draw); the news-increase and engine-buy suppressions take their extra draw only on the branch that applies, so calm-market sequences are unchanged. Everything that happens in the window is logged to a `CrisisEvent` timeline (`CrisisEventType` = `IndustryShock` at trigger, `AuditorRating` for a High/Extra verdict, `Bankruptcy` for a collapse, plus `CompanyClosed` from `CompanyLifecycleService` and `FundClosed` from `CollectiveFundService` when a delisting/fund wind-down lands mid-crisis) — a soft record with no navigations back, read by `GET /crises` (adds `DurationCycles`/`EventCount`) and `GET /crises/{id}` (industries + ordered events) and surfaced on the Crises roster (`CrisesPage`) and standalone `CrisisDetailPage` (`/crises`, `/crises/:id`); the dashboard `CrisisBanner` deep-links to the detail page. `ResetDemoMarketCoreAsync` clears `CrisisEvents` before `Crises` and its sequence alongside them.
- **Share emission dilutes runaway large caps.** `ShareEmissionService` is a supply-side corporate action, not a price move: a company above the $500M capitalisation band rolls each cycle (chance `5% per band`, capped, 50-cycle cooldown) to mint 1–10% new shares and hand them at `AverageCost = 0` to active `Individual`/`AIAgent` non-holders, ≤50 each, via a partial Fisher–Yates over the eligible pool. The per-recipient/holder limits bound the real emission (`min(nominal, eligibleCount × 50)`), so a huge company emits far below its nominal 1–10%. It lists the new shares as a `Filled` zero-price null-participant sell order (the mandated vehicle), raises `IssuedSharesCount` by the distributed amount, records a `ShareEmission`, and posts `Scope = None` news. It stamps no `PriceSnapshot` and cancels no orders — the enlarged supply reprices through ordinary trading. Because it grants holdings while raising the issued count by the same amount, the implicit float (`IssuedSharesCount − Σ Holdings`) is unchanged.
- **Two price-cooldown mechanisms port real-market machinery: a volatility halt and a concentration cap.** `VolatilityHaltService` is a cycle-based Limit Up-Limit Down circuit breaker — deterministic (no `Random`, no chance-rates options, like `StockSplitService`) and run first in the pre-match window so it reads only prior-cycle closes. A live, not-currently-halted company whose latest per-cycle close moved past the up band (`UpBandPercent`) or down band (`DownBandPercent`) versus the close `ObservationWindowCycles` (3) cycles earlier is frozen for `HaltDurationCycles` (15) by stamping `Company.TradingHaltedUntilCycleNumber = currentCycleNumber + duration` — a **cycle number, not an `...InCycleId`**, so the freeze can name a future cycle with no row yet and `MatchingEngine` compares it directly against `cycle.CycleNumber`. The freeze is symmetric (a crash halts like a surge); the down band is set looser than the up band so a single deliberate cut (auditor `Extra` ≤35%, concentration cut 25%) does not self-trip it, and a company trading only on stale history (its latest close older than the window) is never halted. Triggering cancels the **whole** order book — the issuer float and the **human `Player`'s orders included** (buy reservations released) — a new scoped exception to the never-auto-cancel-the-player rule alongside splits/merges and loan distress. While `TradingHaltedUntilCycleNumber >= currentCycleNumber` the company is skipped by `MatchingEngine` (a `continue` in the group loop), excluded from the decision-pass quote universe like a closed company, and rejected by `PlaceOrderCoreAsync` on the non-deferred (player / forced-liquidation) path — the halt simply lapses, no active clearing. `ConcentrationCapService` ports index concentration caps: deterministic, run just before `AuditorService`, it cuts any live company whose worth exceeds `MaxSingleCompanyWeightPercent` (20%) of total market capitalisation by `PriceCutPercent` (25%) via `MarketImpactService.ApplyImpactAsync(Decrease, …)` plus a `Scope = Company` news post — iterative, so a persistent giant is cut again next cycle until it falls under the cap. Both are opt-in (`VolatilityHaltOptions` / `ConcentrationCapOptions`, `Enabled`); `CompanyResponse` and `CompanyDetailResponse` carry an `IsHalted` flag (detail also `HaltedUntilCycleNumber`) surfaced as a market-map tile tag, a detail-page banner, and a disabled `CompanyModal` trade form — the order-book-driven `TradeOrderModal` needs no guard because a halted company has no resting orders to trade against.
- **Collective funds trade with the founder's personality.** A fund is a `Participant` of type `CollectiveFund` whose discretionary trades run through the same `RuleBasedDecisionEngine` as any trader, keyed on the fund participant's own `Temperament` and `RiskProfile`. `CollectiveFundService.OpenFundAsync` snapshots those two fields from the founding participant at creation, so the fund keeps that personality for life even after the founder reprofiles, leaves, or exits the market. The fund's lifecycle machinery (deposit returns, idle/closing sell-downs) uses fixed discounts and does not read temperament or risk.
- **Fund choice, member switching, and founder-driven closure.** When a trader joins, `CollectiveFundService.SelectBestFund` picks the highest-scoring fund among those with room — a min-max-normalised weighted sum of member count, net worth (`FundNetWorth`), and dividends over the last few payout cycles (`RecentDividends`) — replacing the old emptiest-fund rule; ties fall to the lowest fund id. A non-founder member past `MinTenureToSwitchCycles` rolls each cycle (`SwitchChance`, base ± a temperament delta) to leave and chase a better fund; the roll sets `Participant.PendingFundSwitch`, the ordinary leave flow runs, and once the membership is gone the join pass places the member in the best fund ignoring the `JoinBalanceCeiling` and drawing no roll. The founder closes the fund (`MaybeFounderClose`, no random draws) when net worth collapses below `FounderLossFraction` of `CollectiveFund.PeakNetWorth` (ratcheted each cycle) or when `DividendStarved` finds no fund dividend income across its last post-founding payout cycles. `CollectiveFundParticipant.TenureCycles` is incremented once per cycle in the member pass, before the eligibility check. The draw-discipline comment on `ProcessForCycleAsync` is authoritative for scripted-`Random` tests: each member draws at most once (forced-leave *or* switch), and switch-flagged joiners draw nothing. A closed fund is never deleted (a fund is not a `MarketExit` candidate): `FinalizeClose` leaves its `Participant` row as a zeroed, `IsActive = false` tombstone with a `CollectiveFundStatus.Closed` fund row. The `/participants` read hides inactive funds (only funds — bankrupt traders stay listed), so a closed fund drops off the live roster and instead surfaces on the Closed Funds page (`ClosedFundsPage`), fed by the paginated `GET /collective-funds/closed`.
- **Buying on margin: debt is an explicit `Loan`, not a negative balance.** A trader can still reserve for buys beyond its `AvailableBalance`, borrowing up to `DebtLimitFraction` (0.40) of its total worth (cash plus holdings) — but the debt is now carried as open `Loan` rows and `CurrentBalance` is never left negative. When a fill needs more cash than the buyer holds, `LoanService.OriginateLoansForNegativeBalancesAsync` (a post-match sweep in `AdvanceCycleCoreAsync`, run after dividends/shocks so those keep reading pre-match prices, then before the next cycle) turns each resulting negative balance into one `Loan` for `shortfall × (1 + LoanCashBufferRate)` (buffer 0.15), credits the principal so the balance lands at the 15% buffer, and writes a linked `LoanDisbursement` transaction — so `MatchingEngine`'s matching math is untouched. Each loan snapshots the bank's `InterestRatePerCycle` (`0.05 / 50 = 0.001`) and gets a size-scaled term `clamp(25 + (principal / (0.40 × grossWorth)) × 175, 25, 200)`; runtime loans use that mapping while the `AddBanksAndLoans` migration backfills any pre-existing negative balances into fixed 100-cycle loans. `LoanService.ProcessForCycleAsync` (in `MaintainOrdersCoreAsync`, before bankruptcy) charges each open loan its straight-line installment plus interest once per cycle, oldest loan first across a borrower's loans, records `LoanInterest`/`LoanRepayment` transactions, and on a shortfall carries the unpaid amount to next cycle inflated by `MissedPaymentFineRate` (0.10) as `PastDueAmount` with a `LoanFine` record. Inside the final `DistressWindowCycles` (15) of a loan's term, a borrower still in arrears is force-liquidated with below-market sells (`RelatedLoanId`-tagged, deepening discount) reusing the bankruptcy forced-sale mechanics without wiping cash — and this **fires for the human `Player` too**, a deliberate, scoped exception to the otherwise-universal "the market never forces the player" rule, because the player opted into the loan's terms. `DebtAllowanceAsync` and `AvailableCashForDecisions` return remaining loan-principal capacity (`0.40 × worth − open-loan liability`), the latter scaled by `1 / (1 + buffer)` for spendable margin; the buy gate requires `requiredLoanPrincipal = max(0, reserved − AvailableBalance) × (1 + buffer)` to fit that capacity. Net worth everywhere is `CurrentBalance + holdingsValue − openLoanLiability`, and `ParticipantWorthSnapshot.LoanLiability` captures `Σ RemainingPrincipal + Σ PastDueAmount` per cycle. Two counter-pressures still make debt costly, both keyed on loan-liability-as-a-percent-of-worth clamped at the 40% ceiling: `BankruptcyService` adds a flat bankruptcy chance per debt-percent (discharging the loans when it fires), and `RuleBasedDecisionEngine` raises sell willingness per debt-percent by a risk-scaled rate via `DecisionContext.LoanLiability`. A loan closes `PaidInFull` when principal and arrears both reach 0, or `ParticipantDeparted` on bankruptcy / market exit / fund wind-down. `LoanOptions` (config section `Loan`, opt-in `Enabled`) holds the seed bank, rate, term band, buffer, fine, and distress-window tunables; one `Bank` ("National bank") is seeded and also resolved-or-created defensively on first borrow. `GET /banks`, `GET /loans/paged`, `GET /participants/{id}/loans`, and `POST /loans/{id}/repay` back the Banks and Bank-loans pages, the player's repayable loan panel, and the read-only per-trader loan panel.

## Documentation

Human-facing documentation lives in `docs/` and `README.md`. Agent-facing documentation lives in `AGENTS.md` files (this file and per-service variants). Do not cross-link from human docs to `AGENTS.md` — those are for AI agents only.

### Writing Guidelines

- **Avoid specifics that rot.** Do not include library version numbers, file line numbers, or counts of items in a directory. Instead, describe the directory's purpose and point the reader there.
- **Do not duplicate code.** Instead of embedding code snippets that mirror actual source files, point to the file as an example and describe what to look for (function name, section, pattern).
- **Do not enumerate exhaustively.** If a directory contains N files that follow a pattern, describe the pattern and point to the directory — do not list every file.
- **Refer to stable identifiers.** Use function names, class names, section headings, or config keys rather than line numbers. Line numbers go stale on the next commit.
- **Keep human docs and agent docs separate.** `README.md` and `docs/` target developers who will read them in a browser or IDE. `AGENTS.md` files target AI agents that need precise, machine-useful details (exact file paths, exhaustive deviation lists, edge cases). Duplicating content across the two creates a maintenance burden — when something belongs in both, write it in the human docs and reference it from the agent docs, or vice versa, but do not copy it.
- **Do not link to `AGENTS.md` from human-facing docs.** `AGENTS.md` files are for AI agents only and are not meant for human audiences. The only place that should link to an `AGENTS.md` is another `AGENTS.md`. This is a permitted exception to the no-duplication rule — if information exists only in `AGENTS.md` and is also needed in human docs, write a human-appropriate version in `docs/` or `README.md` rather than linking to the agent file.
- **`CLAUDE.md` files are thin redirects.** Every `CLAUDE.md` should contain only a linked reference to the corresponding `AGENTS.md` via `@AGENTS.md`. This ensures compatibility with tools that read `CLAUDE.md` while keeping all substantive content in `AGENTS.md` as the single source of truth.

## Documentation rules

1. Never write documentation that duplicates the information in the `AGENTS.md` and `SUMMARY.md` files.
2. Never write documentation which duplicates what is already written in the code.
3. Documentation should describe features, not implementation details.
4. Documentation should not have references to specific paragraphs in the code.
5. Documentation should not have references to specific files in the code.
6. Never write a code-level identifier (CSS variable, endpoint path, helper, field, route, env var, etc.) into documentation without first verifying it exists verbatim in the source. If a grep does not find it exactly as written, do not write it — a plausible-sounding name is not a verified one.

### Keeping Documentation in Sync

When making changes to the codebase, update documentation in the same PR if the change affects any of the following:

- **New service or top-level directory** — update `README.md` if repository layout description needs it
- **New or changed architectural pattern** — update the Architectural Patterns section in this file. If the pattern is broadly relevant to developers (not just agents), also update `README.md`.
- **New third-party integration** — update the Third-Party Integrations table in `README.md`.
- **Changed local development workflow** — update the Local Development section in `README.md`.
- **New `docs/` page** — add it to the Documentation table in `README.md`.

Documentation updates are not optional extras — they are part of completing the feature.

## Design Context

Frontend design and brand direction live in `PRODUCT.md` at the repo root — read it before changing any UI. Highlights an agent needs up front:

- **Register: `product`.** The frontend is a build-time observability and control dashboard for the simulation, not a marketing surface. Design serves the task.
- **Visual direction: trading-terminal in a light scheme.** Numerics are monospaced and tabular; green and red are reserved for market semantics (up/buy, down/sell) and never used as chrome; ink carries chrome and primary actions.
- **Accessibility bar: WCAG 2.1 AA** — verified text contrast, visible keyboard focus, `prefers-reduced-motion` alternatives, and no color-only signals (buy/sell and up/down also carry text or a glyph).
- **Design tokens** (OKLCH palette, fixed `rem` type scale, spacing) live in `frontend/src/index.css`; component styles in `frontend/src/App.css`. Reuse the tokens. Anti-references to avoid: warm-cream SaaS palettes, rounded ghost-cards with wide drop shadows, marketing-scale hero headings, and per-section uppercase eyebrows.
